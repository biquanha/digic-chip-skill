# 参考模型与计分板设计

本文用于指导什么时候需要 reference model，什么时候简单直通比较就够，以及 scoreboard 如何拆层。

## 什么时候需要 reference model

以下场景通常需要参考模型：
- DUT 有算法变换，不是简单透传
- DUT 有寄存器配置影响输出行为
- DUT 存在重排序、仲裁、缓冲
- DUT 输出与输入不是一一同拍对应
- DUT 带状态机，结果依赖历史事务

以下场景可先用最小预测器：
- 简单寄存器透传
- 固定延迟流水线
- 单输入单输出、规则简单的变换

## 推荐拆分

### predictor / reference model
负责：
- 接收输入事务
- 根据配置和协议规则预测期望输出
- 发给 scoreboard

### scoreboard
负责：
- 缓存 expected / actual
- 做配对
- 比较字段
- 统计 compare_count / error_count
- 输出摘要

## 真实项目里的常见坑

- 只比较 payload，不比较 sideband
- 没考虑顺序变化，导致误报
- reset 后 expected 队列未清空
- monitor 粒度不一致，导致 expected/actual 无法配对
- 参考模型忽略配置寄存器，导致比较永远错

## 最小实现建议

第一版先做到：
- expected 与 actual 结构统一
- scoreboard 有明确配对策略
- report_phase 输出摘要
- reset 时清空缓存

第二版再扩展：
- 多通道缓存
- tag/id 配对
- out-of-order 配对
- sideband 字段比较
