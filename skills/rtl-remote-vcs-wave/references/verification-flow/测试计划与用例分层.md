# 测试计划与用例分层

本文用于指导 agent 在面对一个新的 DUT 时，如何把“先跑通”升级为“可扩展、可回归、可度量”的验证计划。

## 目标

对于新模块，不要只给一个 smoke case。
应至少规划以下几层 testcase：
- smoke：验证最小主路径可跑通
- basic：覆盖常规功能组合
- cfg：覆盖寄存器配置与模式切换
- error：覆盖非法输入、边界值、异常响应
- reset：覆盖复位打断、复位恢复、上电初态
- stress：覆盖长时间随机流量与高负载
- interleave：覆盖多通道/多事务交织
- regression：挑选一组可日常回归的核心 case

## 渐进式拆分方法

### 第 1 阶段：先建 smoke

smoke 的目标不是“覆盖全”，而是验证：
- 编译链路打通
- interface/driver/monitor 接线正确
- 至少有一个事务进入 DUT
- monitor 能观测到输出
- scoreboard 能完成一次成功比较
- FSDB 能生成

### 第 2 阶段：补 basic

basic case 通常按 DUT 的主要功能点拆分：
- 不同操作类型
- 不同配置模式
- 典型输入组合
- 正常时序与轻微 backpressure

### 第 3 阶段：补 error 与 reset

这是很多最小平台容易漏掉的部分。
应明确：
- 非法输入后 DUT 预期行为
- 配置错误时 DUT 预期响应
- reset 期间事务是否应丢弃
- reset 后内部状态是否清零
- monitor/scoreboard 是否需要 reset-aware 处理

### 第 4 阶段：补 stress 与 interleave

当 DUT 带有队列、仲裁、多端口、多通道时，这部分很关键：
- 长包与短包混合
- 通道间交织
- 持续 backpressure
- 突发流量
- 接近 FIFO 深度上限

## 推荐的 testcase 命名

建议统一命名，便于回归和日志筛选：
- `smoke_test`
- `basic_rw_test`
- `cfg_mode_switch_test`
- `error_illegal_cmd_test`
- `reset_during_busy_test`
- `stress_random_1k_test`
- `interleave_multi_chan_test`

## 每类 testcase 需要明确的 5 个问题

agent 应主动追问或补齐：
- 这个 case 要验证哪个功能点？
- 输入激励从哪里来，约束是什么？
- 参考结果如何生成？
- pass/fail 判据是什么？
- 是否需要覆盖率采样与额外日志？

## 回归集规划建议

### 日常回归
- 1 个 smoke
- 2~4 个 basic
- 1 个 reset
- 1 个 error

### 周回归
- 补充 stress
- 补充 interleave
- 补充随机种子回归

### 里程碑回归
- 全 case 集合
- 多 seed 扩展
- 覆盖率收敛检查
- 关键波形 spot check
