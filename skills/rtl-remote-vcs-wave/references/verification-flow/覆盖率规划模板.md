# 覆盖率规划模板

本文用于指导新模块的 coverage 规划，避免只有 testcase 数量，没有覆盖闭环。

## 覆盖率分层

建议至少拆成三层：
- code coverage：行、条件、FSM、toggle
- functional coverage：功能场景是否被验证到
- cross coverage：关键维度组合是否被覆盖到

## 功能覆盖规划步骤

### 1. 先识别关键维度

常见关键维度包括：
- 操作类型
- 输入长度/大小
- 通道号
- 优先级
- 配置模式
- backpressure 状态
- reset 插入位置
- 错误类型

### 2. 把维度映射成 coverpoint

例如：
- `cmd_type`
- `data_len`
- `channel_id`
- `mode_sel`
- `rsp_status`

### 3. 只对真正有价值的组合做 cross

不要一开始就把所有项全 cross。
优先 cross：
- 操作类型 x 配置模式
- 操作类型 x 响应状态
- 通道号 x backpressure
- reset 插入位置 x 事务阶段

## 覆盖率评审时要问的问题

- 这个 coverpoint 是否对应验证计划中的功能点？
- 是否有 unreachable bin？
- 是否缺关键异常场景的 bin？
- 覆盖率增长是否真正代表功能被验证，而不是随机碰撞？

## 推荐输出

当用户要求做覆盖规划时，agent 应至少输出：
- 覆盖维度列表
- 每个维度的 coverpoint 草案
- 关键 cross 列表
- 哪些 bin 由 directed case 负责
- 哪些 bin 由 random + constraint 负责
